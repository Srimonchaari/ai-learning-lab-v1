Architecture decisions

Deterministic DAG traversal over ML ranking System Design

I chose topological traversal of the prerequisite graph over a scoring model. ML ranking requires engagement signals that don't exist at V1. Deterministic traversal is auditable, testable, and produces identical output for identical state — critical properties for a learning system where order matters.

Three-tier fallback prevents silent failure Resilience

RSS feeds fail unpredictably. A hard failure when strict feeds return zero results would block the entire daily loop. The fallback chain widens the source pool (strict → category-broader → global AI feeds) while keeping the concept node fixed. Crucially, each tier logs its strategy name — observable, not silent.

Keyword regex over embedding similarity Evaluation

Regex word-boundary matching is fast, deterministic, and interpretable. Embedding similarity adds API cost, latency, and a non-deterministic component. The upgrade criterion is explicit: when measured keyword recall degrades below an acceptable threshold across multiple concept categories.

Flat JSON state over a database Simplicity

Single-user V1 reads the entire progress state on every run. A database adds connection management, schema migrations, and query complexity for a file that never has concurrent writers. Flat JSON is portable, version-controllable, and auditable. Migration path to a DB is cheap when multi-user support arrives.

Key algorithm — prerequisite-aware selection (roadmap.py)
def select_next_concept(
    concepts: List[Dict],
    completed: List[str]
) -> Dict:
    for concept in concepts:
        cid     = concept["id"]
        prereqs = concept.get("prereqs", [])

        if cid in completed:
            continue  # already done

        if all(p in completed for p in prereqs):
            return concept  # first eligible node

    raise RuntimeError(
        "No eligible concept found."
    )

Engineering approach

How I think about AI systems

Four principles that shape every architecture decision I make — from a solo CLI tool to a production-scale pipeline.

01

Start with the evaluation question

Before choosing an architecture, I ask: how will I know if this is working? A system without a defined evaluation method is a prototype, not a product. For Synapz, the evaluation question was: "Does the selected content match the active learning concept?" That determined the filtering approach — not technology preference.

Applied in Synapz: Chose keyword regex over embeddings because it made precision measurable and failures visible.

Match complexity to evidence

02

I don't add vector stores, rerankers, or LLM layers until simpler approaches demonstrably fail. Every added component creates a new failure surface. The right architecture is the simplest one that satisfies the current evaluation constraints — with a clear criterion for when to upgrade.

Applied in Synapz: Flat JSON over SQLite. RSS over web scraping. Regex over embeddings. Each chosen after evaluating the real trade-off.
03

Make trade-offs explicit and reversible

Every design choice gives something up. I document what was chosen and why, and — critically — define the condition under which the decision should be revisited. Implicit trade-offs become untracked technical debt. Explicit ones become architecture decisions you can defend in a review.

Applied in Synapz: Each major choice has a "revisit when" criterion — trigger for upgrading, not just a note that upgrading is possible.
04

Design for observable failure

AI systems fail silently — empty results, degraded quality, mismatched content — without raising exceptions. I design observability in from the start: structured logging at every layer, strategy names logged with each output, and fallback tiers that surface which path fired and why.

Applied in Synapz: The fallback strategy logs strategy=category_primary, category_fallback, or global_fallback on every run — so degradation is never invisible.

Synapz — AI Learning Orchestration Engine

A prerequisite-aware curriculum engine that ingests curated technical content and delivers one structured daily learning unit. Designed to enforce curriculum integrity — concepts unlock only when prerequisites are met, and retrieved content is always anchored to the active concept's keyword space.

Problem

Learners jump between disconnected topics without prerequisite ordering, producing knowledge gaps that block production-level work.

Solution

A DAG-based selection engine over a YAML curriculum, with category-aware feed ingestion and three-tier fallback to prevent silent failures.

Key outcome

Every run produces a deterministic, reproducible output — same state always yields the same concept and the same source pool logic.

Evaluation design

Precision measurable via keyword match rate. Strategy tier logged per run enables degradation tracking. Unit tests cover all three modules.

Technical highlights:
  • Frozen FeedItem dataclass for immutable feed entries
  • Word-boundary regex prevents false positives (e.g. "rage" ≠ "rag")
  • URL seen-set deduplication — not title hashing — handles cross-feed duplicates
  • Structured logging at every pipeline stage with strategy attribution
  • Full pytest suite with offline RSS fixtures for deterministic source tests
