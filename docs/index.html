Featured System

Synapz — AI Learning Orchestration Engine

A CLI-driven pipeline that converts a prerequisite graph and curated RSS feeds into a structured daily learning unit. Built to demonstrate how curriculum integrity, content ingestion, and relevance filtering compose into a reliable system.

Runtime pipeline

  1. 01
    Learner state + roadmap

    Load progress JSON and YAML curriculum. Traverse the prerequisite DAG to select the next eligible concept.

  2. 02
    Feed ingestion

    Fetch from curated RSS packs (arXiv, AWS ML, HuggingFace). Age-filter by configurable window, cap entries per feed.

  3. 03
    Relevance filter

    Regex keyword matching on title and summary. URL deduplication via seen-set. Three-tier fallback: strict → broader → global.

  4. 04
    Daily unit output

    Concept brief, mini project, interview challenge — one structured output per run.

Design decisions

  • Deterministic selection over ML ranking

    Topological DAG traversal is more debuggable and auditable than a scoring model before engagement signals exist.

  • Keyword matching over embeddings

    Fast, interpretable, zero infrastructure. The upgrade point is when keyword precision demonstrably degrades — not before.

  • Flat JSON state, not a database

    Portable, auditable, version-controllable. Migration cost at scale is low and the complexity isn't warranted at V1.

  • Three-tier feed fallback

    Prevents silent zero-result failures without compromising curriculum integrity. Concept node stays fixed; source pool widens.

Engineering approach

How I think about AI systems

Start with the evaluation question

Before choosing an architecture, I ask: how will I know if this is working? A system without a clear evaluation method is a prototype, not a product. I define success criteria — precision, latency, failure modes — before writing the first line of retrieval or generation code.

Match complexity to evidence

I don't add vector stores, embedding models, or rerankers until keyword matching demonstrably fails. Every layer of complexity adds failure surface. The right architecture is the simplest one that satisfies the current evaluation constraints.

Make trade-offs explicit

Every design choice trades something. I document what was accepted and why — deterministic vs. probabilistic, latency vs. recall, simplicity vs. coverage. Explicit trade-offs are reviewable. Implicit ones become technical debt.

Build for debuggability

AI systems fail silently. I design for observability from the start: structured logging, deterministic seed behaviors, single-responsibility modules, and fallback paths that surface which tier fired and why.
